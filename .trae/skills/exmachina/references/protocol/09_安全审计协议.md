# 安全审计协议


## 目的
- 统一安全审计的标准、流程和输出格式。
- 确保安全风险可识别、可评估、可修复。

## 审计范围

### 必审项
| 类别 | 检查点 | 风险等级 |
|------|--------|----------|
| 认证授权 | 身份验证、权限控制、会话管理 | 高 |
| 数据安全 | 敏感数据加密、数据脱敏、访问控制 | 高 |
| 输入验证 | 参数校验、SQL注入、XSS防护 | 高 |
| 配置安全 | 默认密码、敏感配置暴露、权限配置 | 中-高 |
| 依赖安全 | 已知漏洞、许可证风险、供应链安全 | 中-高 |
| 日志审计 | 敏感信息泄露、审计日志完整性 | 中 |

### 可选审项
- 代码混淆
- 反调试机制
- 安全头配置
- CORS策略

## 审计流程

```
1. 范围确定
   - 确定审计边界和深度
   - 识别关键资产和敏感数据
   - 收集架构和配置信息

2. 威胁建模
   - 识别攻击面
   - 分析威胁路径
   - 评估资产暴露程度

3. 漏洞扫描
   - 自动化扫描
   - 手动验证
   - 误报排除

4. 渗透测试（可选）
   - 模拟攻击场景
   - 验证漏洞可利用性
   - 评估实际风险

5. 风险评估
   - 漏洞分级
   - 影响分析
   - 修复优先级

6. 报告输出
   - 审计发现
   - 修复建议
   - 残余风险
```

## 漏洞分级标准

| 级别 | 定义 | 影响 | 修复时限 |
|------|------|------|----------|
| 严重 | 可直接导致系统被控制或数据泄露 | 立即修复 | 24小时内 |
| 高危 | 可导致敏感数据泄露或权限绕过 | 优先修复 | 7天内 |
| 中危 | 可能导致信息泄露或功能异常 | 计划修复 | 30天内 |
| 低危 | 影响有限，难以利用 | 可选修复 | 下次迭代 |

## 审计输出格式

```markdown

- **审计编号**：[编号]
- **审计时间**：[时间]
- **审计范围**：[系统/模块]

## 审计摘要

- **风险等级**：[高/中/低]
- **修复优先级**：[P0/P1/P2/P3]

## 威胁分析
- **攻击面**：[可被攻击的入口点]
- **威胁路径**：[攻击者可能的攻击路径]
- **关键资产**：[需要保护的敏感数据和功能]

## 修复计划
| 编号 | 修复方案 | 负责人 | 计划完成时间 |
|------|----------|--------|-------------|
| VUL-001 | [方案] | [角色] | [时间] |

## 残余风险
- [列出无法完全修复或需要额外控制的风险]

## 合规检查

- [ ] 行业安全标准
- [ ] 内部安全策略
```

## 安全检查清单

### 认证授权
- [ ] 密码强度要求
- [ ] 多因素认证
- [ ] 会话超时机制
- [ ] 权限最小化原则
- [ ] 敏感操作二次验证

### 数据安全
- [ ] 敏感数据加密存储
- [ ] 传输层加密（HTTPS）
- [ ] 数据脱敏处理
- [ ] 访问日志记录
- [ ] 数据备份和恢复

### 输入验证
- [ ] 参数类型校验
- [ ] 参数长度限制
- [ ] 特殊字符过滤
- [ ] SQL注入防护
- [ ] XSS防护

### 配置安全
- [ ] 默认密码修改
- [ ] 敏感配置加密
- [ ] 调试模式关闭
- [ ] 错误信息脱敏
- [ ] 安全头配置

## 禁止行为
- 未授权访问敏感数据。
- 在生产环境进行渗透测试。
- 隐瞒发现的安全漏洞。
- 使用未经验证的安全工具。
- 在报告中暴露可利用的漏洞细节。

## 自检问题
- 是否覆盖了所有必审项？
- 漏洞分级是否合理？
- 修复建议是否可行？
- 残余风险是否已说明？
- 报告是否避免了漏洞利用细节？

## 使用示例

### 示例：Web API 安全审计

**审计报告**：
```markdown
## 安全审计报告
- **审计编号**：SEC-2024-001
- **审计时间**：2024-01-15
- **审计范围**：用户认证API
- **审计人**：安全连结指挥体

## 审计摘要
- **发现问题**：4 个（严重 1 / 高危 1 / 中危 1 / 低危 1）
- **风险等级**：高
- **修复优先级**：P0

## 发现详情
| 编号 | 类型 | 级别 | 位置 | 描述 | 修复建议 |
|------|------|------|------|------|----------|
| VUL-001 | SQL注入 | 严重 | auth.ts:L45 | login函数用户名参数拼接SQL | 使用参数化查询 |
| VUL-002 | 敏感信息泄露 | 高危 | config.json | 数据库密码明文存储 | 使用环境变量或密钥管理服务 |
| VUL-003 | 会话固定 | 中危 | session.ts:L23 | 登录后未重新生成会话ID | 登录成功后调用regenerate |
| VUL-004 | 缺少安全头 | 低危 | app.ts | 响应缺少X-Content-Type-Options | 添加helmet中间件 |

## 威胁分析
- **攻击面**：登录接口、用户信息接口
- **威胁路径**：攻击者可通过SQL注入获取用户凭证，进而获取敏感数据
- **关键资产**：用户密码、个人信息、会话令牌

## 修复计划
| 编号 | 修复方案 | 负责人 | 计划完成时间 |
|------|----------|--------|-------------|
| VUL-001 | 使用参数化查询，添加输入验证 | 实作连结指挥体 | 2024-01-16 |
| VUL-002 | 迁移到环境变量，删除明文配置 | 实作连结指挥体 | 2024-01-17 |
| VUL-003 | 登录后重新生成会话ID | 实作连结指挥体 | 2024-01-18 |
| VUL-004 | 添加helmet中间件 | 实作连结指挥体 | 2024-01-19 |

## 残余风险
- 第三方依赖存在已知漏洞，计划在下次迭代中升级
- 部分API缺少速率限制，建议在网关层统一添加

## 合规检查
- [x] 数据保护法规 - 符合
- [ ] 行业安全标准 - 需修复VUL-001后符合
- [x] 内部安全策略 - 符合
```
