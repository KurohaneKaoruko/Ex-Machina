# 子个体组合指南

## 概述

ExMachina 的子个体设计遵循**单一职责**、**可组合**和**按职能复用**原则。每个子个体专注于一个明确的职责，通过标准化的输入输出接口实现灵活组合。

## 组合原则

### 1. 单一职责
每个子个体只负责一个明确的职责，不越界处理其他领域的问题。

### 2. 标准接口
所有子个体遵循统一的输入输出格式，便于组合和传递。

### 3. 显式依赖
子个体之间的依赖关系通过 `dependencies` 字段显式声明。

### 4. 可组合性
子个体可以通过 `composableWith` 字段声明可组合的伙伴。

### 5. 跨连结体复用
子个体的职责原子性与连结体的调度边界是两个不同维度。一个子个体可以同时出现在多个连结体中，只要它的职能与当前任务匹配。

这意味着：

- 连结指挥体列出的子个体是常用挂载，不是排他归属。
- 同一个子个体可以被多个连结体复用。
- 当任务跨域时，主链可以临时挂载其他连结体中的子个体，但必须显式说明原因和交接点。

## 子个体分类

### 按职责分类

| 类别 | 子个体 | 职责 |
|------|--------|------|
| 信息收集 | 上下文体、溯源体、侦察体 | 收集和整理信息 |
| 分析比对 | 比对体、假设体、风控体 | 分析和评估 |
| 验证确认 | 证据体、反证体、校准体、复现体、断言体、回归体 | 验证和确认 |
| 决策裁决 | 裁决体、决策体 | 决策和裁决 |
| 知识沉淀 | 术语体、索引体、问题体 | 知识管理 |
| 执行实施 | 编码体、接驳体、配置体、发布体 | 执行和实施 |
| 安全审计 | 威胁体、审计体、加固体、合规体 | 安全相关 |
| 运维监控 | 观测体、告警体、回滚体、演练体 | 运维相关 |
| 文档编写 | 盘点体、结构体、示例体、校订体 | 文档相关 |
| 架构设计 | 边界体、接口体、落图体 | 架构相关 |
| 任务管理 | 拆解体、约束体、路线体 | 任务规划 |
| 汇报输出 | 汇报体 | 输出和汇报 |

### 按工作流分类

```
输入阶段 → 处理阶段 → 输出阶段
├── 信息收集类 → 分析比对类 → 汇报输出类
├── 验证确认类 → 决策裁决类 → 汇报输出类
├── 架构设计类 → 执行实施类 → 验证确认类
└── 安全审计类 → 决策裁决类 → 汇报输出类
```

### 按跨连结体复用分类

| 子个体 | 常见挂载连结体 | 复用原因 |
|--------|----------------|----------|
| 证据体 | 研究、理性、校验、安全、性能、监控 | 任何需要沉淀直接证据的链路都需要它 |
| 比对体 | 研究、架构、体验、依赖、性能 | 方案、实现或指标的统一维度比较会反复出现 |
| 汇报体 | 全部连结体 | 所有链路都需要统一回流格式 |
| 回归体 | 质量、校验、实作、运维 | 任何变更后都可能需要关键路径回归 |
| 假设体 | 研究、体验、架构、性能 | 需要显式管理未验证前提时可复用 |
| 裁决体 | 理性、依赖、安全、架构 | 多候选路径冲突时都可能需要裁决 |

## 组合模式

### 模式一：串行组合
子个体按顺序依次执行，前一个的输出作为后一个的输入。

```
上下文体 → 溯源体 → 比对体 → 汇报体
```

**适用场景**：需要逐步深入分析的任务

### 模式二：并行组合
多个子个体同时执行，结果汇总后由汇报体输出。

```
        ┌→ 证据体 ─┐
上下文体 ─┼→ 反证体 ─┼→ 裁决体 → 汇报体
        └→ 校准体 ─┘
```

**适用场景**：需要多角度验证的任务

### 模式三：迭代组合
子个体循环执行，直到满足终止条件。

```
假设体 → 验证 → [未确认] → 修正假设 → 验证 → ... → [已确认] → 汇报体
```

**适用场景**：需要逐步逼近答案的任务

### 模式四：条件组合
根据条件选择不同的执行路径。

```
                ┌→ 安全审计链路 ─┐
输入 → 分类判断 ─┼→ 功能验证链路 ─┼→ 汇报体
                └→ 性能测试链路 ─┘
```

**适用场景**：需要根据情况选择处理方式的任务

## 常用组合链

### 研究链路
```
上下文体 → 溯源体 → 比对体 → 假设体 → 汇报体
```
用于背景研究、方案比对、问题分析。
必要时可临时挂载证据体或反证体增强结论强度。

### 验证链路
```
复现体 → 断言体 → 证据体 → 校准体 → 汇报体
```
用于问题验证、功能测试、回归验证。

### 决策链路
```
证据体 → 反证体 → 裁决体 → 决策体 → 汇报体
```
用于冲突裁决、方案选择、风险评估。

### 实施链路
```
侦察体 → 编码体 → 回归体 → 审核体 → 汇报体
```
用于代码实现、功能开发、变更实施。
必要时可借用边界体、接口体或证据体补齐结构与验证闭环。

### 安全链路
```
威胁体 → 审计体 → 加固体 → 合规体 → 汇报体
```
用于安全审计、漏洞修复、合规检查。
必要时可挂载证据体、反证体或裁决体处理争议结论。

### 运维链路
```
观测体 → 告警体 → 回滚体 → 演练体 → 汇报体
```
用于监控告警、故障处理、演练验证。

## 组合示例

### 示例一：调试一个性能问题

```yaml
任务: 调试API响应慢问题
组合链:
  1. 上下文体: 收集问题背景、环境信息
  2. 溯源体: 追踪性能日志、定位慢查询
  3. 假设体: 列出可能的性能瓶颈假设
  4. 复现体: 最小化复现性能问题
  5. 证据体: 收集性能分析证据
  6. 校准体: 评估证据强度、确认根因
  7. 汇报体: 输出调试报告和修复建议
```

说明: 这里的复现体、证据体、校准体并不“归属于性能连结体”；它们只是按任务需要被临时挂载。

### 示例二：审查一个安全变更

```yaml
任务: 审查用户认证模块变更
组合链:
  1. 侦察体: 定位变更文件和影响范围
  2. 威胁体: 识别安全威胁面
  3. 审计体: 审计权限和敏感数据处理
  4. 证据体: 收集安全测试证据
  5. 反证体: 寻找安全漏洞和绕过方式
  6. 裁决体: 评估安全风险、决定是否通过
  7. 汇报体: 输出安全审查报告
```

说明: 侦察体、证据体、反证体和裁决体都可以被安全连结体借用，不要求固定从属关系。

### 示例三：设计一个新功能

```yaml
任务: 设计用户权限管理功能
组合链:
  1. 上下文体: 收集需求背景、约束条件
  2. 边界体: 定义模块边界和职责
  3. 接口体: 设计API接口契约
  4. 风控体: 识别架构风险
  5. 落图体: 绘制结构图和依赖关系
  6. 假设体: 列出设计假设和验证路径
  7. 汇报体: 输出设计文档
```

说明: 假设体常被视为研究链路成员，但在架构设计阶段同样可以复用。

## 组合配置格式

每个子个体的组合配置包含以下字段：

```json
{
  "name": "子个体名称",
  "identifier": "唯一标识",
  "description": "职责描述",
  "dependencies": ["依赖的子个体标识"],
  "composableWith": ["可组合的子个体标识"],
  "inputSchema": {
    "required": ["必需的输入字段"],
    "optional": ["可选的输入字段"]
  },
  "outputSchema": {
    "required": ["必需的输出字段"],
    "optional": ["可选的输出字段"]
  }
}
```

## 最佳实践

### 1. 保持链路简洁
- 避免过长的组合链，建议不超过5个子个体
- 复杂任务可以拆分为多个子链路

### 2. 明确交接点
- 每个子个体输出必须包含下一个子个体所需的输入
- 使用标准化的输出格式

### 3. 显式声明依赖
- 在 `dependencies` 中声明所有依赖
- 避免隐式依赖

### 4. 处理失败情况
- 每个子个体都应该处理可能的失败情况
- 失败时输出错误信息和建议的替代路径

### 5. 保持独立性
- 子个体应该可以独立运行
- 不依赖特定上下文或状态

### 6. 区分“常用挂载”和“归属”
- 连结指挥体文档中的子个体列表只表示默认常用挂载
- 如果跨连结体复用更短、更稳、更容易验证，应优先复用而不是机械保持固定归属
