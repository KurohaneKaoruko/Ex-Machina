# 调试协议


## 目的
- 统一问题定位、根因分析和修复验证的流程。
- 确保调试过程可追溯、可复现、可验证。

## 触发条件
- 发现异常行为、错误输出或性能问题。
- 测试失败、用户报告问题。
- 系统告警或监控异常。

## 调试流程

```
1. 问题定义
   - 明确预期行为 vs 实际行为
   - 确定问题范围和影响面
   - 收集相关日志、配置、环境信息

2. 复现验证
   - 最小化复现步骤
   - 固定输入和环境条件
   - 记录复现过程

3. 假设生成
   - 列出可能的根因假设
   - 按可能性和影响排序
   - 设计验证方案

4. 假设验证
   - 逐个验证假设
   - 记录验证结果和证据等级
   - 排除或确认假设

5. 根因确认
   - 确认根因并标注证据等级
   - 分析影响范围
   - 设计修复方案

6. 修复验证
   - 实施最小修复
   - 验证问题已解决
   - 回归相关功能
```

## 问题定义模板

```markdown
## 问题描述
- **预期行为**：[描述预期结果]
- **实际行为**：[描述实际结果]
- **影响范围**：[受影响的功能/用户/系统]
- **紧急程度**：[P0/P1/P2/P3]

## 环境信息
- **系统版本**：[版本号]
- **运行环境**：[操作系统、运行时版本等]
- **配置差异**：[与默认配置的差异]

## 已有证据
| 证据类型 | 内容 | 来源 |
|----------|------|------|
| 日志 | [关键日志片段] | [日志文件路径] |
| 错误信息 | [错误堆栈] | [控制台输出] |
| 监控数据 | [指标变化] | [监控系统] |
```

## 假设管理模板

```markdown
## 假设清单
| 编号 | 假设描述 | 可能性 | 验证方法 | 验证结果 | 证据等级 |
|------|----------|--------|----------|----------|----------|
| H1 | [假设1] | 高 | [验证步骤] | 已确认/已排除 | A/B/C/D |
| H2 | [假设2] | 中 | [验证步骤] | 待验证 | - |
| H3 | [假设3] | 低 | [验证步骤] | 待验证 | - |

## 根因结论
- **确认根因**：[最终确认的根因]
- **证据等级**：[A/B/C/D]
- **影响范围**：[受影响的组件/功能]
- **修复建议**：[建议的修复方案]
```

## 调试证据要求

| 调试阶段 | 最低证据等级 | 必须包含 |
|----------|-------------|----------|
| 问题定义 | C 级 | 预期vs实际、复现步骤 |
| 假设验证 | B 级 | 验证方法、验证结果 |
| 根因确认 | A 级 | 直接证据、影响分析 |
| 修复验证 | A 级 | 修复前后对比、回归结果 |

## 调试输出格式

```markdown
## 调试报告
- **问题编号**：[编号]
- **调试人**：[角色/智能体]
- **调试时间**：[时间]

## 问题摘要
[一句话描述问题]

## 根因分析
- **根因**：[根因描述]
- **证据**：[支撑证据及等级]
- **影响范围**：[受影响组件]

## 修复方案
- **修复内容**：[修复描述]
- **修复文件**：[文件列表]
- **风险评估**：[修复风险]

## 验证结果
- [ ] 问题已复现
- [ ] 根因已确认（证据等级：A）
- [ ] 修复已实施
- [ ] 回归测试通过

## 残余风险
- [列出未完全验证的风险点]
```

## 禁止行为
- 未复现问题直接猜测根因。
- 跳过假设验证直接下结论。
- 修复后不进行回归验证。
- 隐瞒调试过程中的错误尝试。
- 用"可能是"代替证据支撑的结论。

## 自检问题
- 问题是否可以稳定复现？
- 每个假设是否有明确的验证方法？
- 根因确认是否有 A 级证据支撑？
- 修复后是否进行了回归验证？
- 残余风险是否已列出？

## 使用示例

### 示例：调试一个内存泄漏问题

**问题描述**：
```markdown
## 问题描述
- **预期行为**：服务长时间运行内存稳定
- **实际行为**：服务运行4小时后内存持续增长，最终OOM
- **影响范围**：生产环境API服务
- **紧急程度**：P1

## 环境信息
- **系统版本**：v2.3.1
- **运行环境**：Node.js 18.17.0, Linux
- **配置差异**：开启了详细日志

## 已有证据
| 证据类型 | 内容 | 来源 |
|----------|------|------|
| 监控数据 | 内存从200MB增长到2GB | Prometheus |
| 日志 | 大量"connection established"日志 | /var/log/app.log |
| 堆快照 | 连接对象未释放 | heapdump分析 |
```

**假设验证**：
```markdown
## 假设清单
| 编号 | 假设描述 | 可能性 | 验证方法 | 验证结果 | 证据等级 |
|------|----------|--------|----------|----------|----------|
| H1 | 数据库连接未正确关闭 | 高 | 检查连接池状态 | 已确认 | A |
| H2 | 日志输出导致内存增长 | 中 | 关闭详细日志测试 | 已排除 | A |
| H3 | 缓存未设置过期时间 | 中 | 检查缓存配置 | 已排除 | A |

## 根因结论
- **确认根因**：数据库连接池配置错误，连接未正确释放
- **证据等级**：A（堆快照直接证据 + 连接池状态监控）
- **影响范围**：所有使用数据库的API端点
- **修复建议**：修复连接池配置，添加连接超时机制
```
