# 代码审查协议


## 目的
- 统一代码审查的标准、流程和输出格式。
- 确保审查结论可追溯、可验证、可行动。

## 触发条件
- 任何代码变更在合并前必须经过审查。
- 关键路径、安全相关、性能敏感的变更需要更严格的审查。

## 审查维度

### 必审维度
| 维度 | 检查点 | 证据等级要求 |
|------|--------|-------------|
| 正确性 | 逻辑是否正确、边界条件是否覆盖 | A 级：测试结果、运行验证 |
| 安全性 | 是否引入安全风险、敏感数据处理 | A 级：代码审查、安全扫描 |
| 性能 | 是否影响性能、资源使用是否合理 | B 级：性能测试、代码分析 |
| 可维护性 | 代码结构、命名、注释是否清晰 | B 级：代码审查 |
| 兼容性 | 是否破坏现有接口、依赖是否兼容 | A 级：测试结果 |

### 可选维度
- 文档完整性
- 测试覆盖率
- 错误处理完整性

## 审查流程

```
1. 收集变更上下文
   - 变更范围、影响面、相关依赖
   - 现有测试、文档、配置

2. 逐维度审查
   - 按优先级检查各维度
   - 记录发现的问题和证据

3. 问题分级
   - P0 阻断：必须修复才能合并
   - P1 重要：建议修复，可后续跟进
   - P2 建议：可选优化

4. 输出审查结论
   - 审查结果：通过/有条件通过/不通过
   - 问题清单：分级列出
   - 建议修复方案
```

## 审查输出格式

```markdown
## 审查结论
- 审查结果：[通过/有条件通过/不通过]
- 审查范围：[文件列表]
- 审查人：[角色/智能体]

## 发现问题
| 编号 | 维度 | 级别 | 描述 | 证据 | 建议修复 |
|------|------|------|------|------|----------|
| #1 | 安全性 | P0 | SQL注入风险 | 文件A第10行 | 使用参数化查询 |

## 审查通过条件
- [ ] P0 问题已全部修复
- [ ] P1 问题已有跟进计划
- [ ] 测试全部通过

## 残余风险
- [列出已知但本次未覆盖的风险点]
```

## 审查证据要求

| 问题级别 | 最低证据等级 | 必须包含 |
|----------|-------------|----------|
| P0 阻断 | A 级 | 代码位置、问题描述、复现步骤或测试用例 |
| P1 重要 | B 级 | 代码位置、问题描述、影响分析 |
| P2 建议 | C 级 | 问题描述、建议方向 |

## 禁止行为
- 未审查直接通过。
- 只看代码不运行测试。
- 隐瞒发现的问题。
- 用"看起来没问题"代替证据支撑的结论。
- 审查时引入与变更无关的新问题。

## 自检问题
- 是否覆盖了所有必审维度？
- 每个问题是否有明确的证据支撑？
- 问题分级是否合理？
- 是否说明了残余风险？
- 审查结论是否可行动？

## 使用示例

### 示例：审查一个 API 变更

**输入**：
```
变更内容：修改用户登录接口，增加验证码校验
变更文件：src/api/auth.ts, src/utils/captcha.ts
```

**审查输出**：
```markdown
## 审查结论
- 审查结果：有条件通过
- 审查范围：src/api/auth.ts (L45-78), src/utils/captcha.ts (全文)
- 审查人：审核体

## 发现问题
| 编号 | 维度 | 级别 | 描述 | 证据 | 建议修复 |
|------|------|------|------|------|----------|
| #1 | 安全性 | P0 | 验证码未设置过期时间 | captcha.ts L23 | 添加5分钟过期逻辑 |
| #2 | 性能 | P1 | 验证码校验未做频率限制 | auth.ts L52 | 添加 rate limiting |
| #3 | 可维护性 | P2 | 缺少错误处理文档 | auth.ts L60-70 | 补充错误码说明 |

## 审查通过条件
- [x] P0 问题已全部修复 (待验证)
- [ ] P1 问题已有跟进计划
- [x] 测试全部通过

## 残余风险
- 验证码存储方案在高并发场景下未验证
- 第三方验证码服务的可用性依赖未评估
```
